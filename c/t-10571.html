Hi,
let me make one thing clear. a Webservice is NOT a web application. Many people misunderstand this term, as think it is a type of webapplication.
but, webservice can be described as, "accessing any business logic on Internet", in lay man's terms.
to be short, a client application, sends a XML file,(in a standard format, which we call as SOAP) which is received by a server application (or a service provider), and it invokes any business processes and then sends response back to the user.(which is again by sending an XML file). the client can use the results.

In ur scenario, when we create a webservice for a baan customer, suppliers can invoke this service sitting in their own system. lets say, the customers have SAP. Sitting inside SAP, they can invoke our webservice. and submit the shipment details.

As far as making a webservice for baan is concerned, we can do it, either by Openworld or BCK. as making a webservices needs, XML support(WSDL, SOAP, UDDI), which baan doesnt give directly, but possible by external application connectivity with baan.

If we want the customer to go to a website, and give the shipment details, then any webapplication will solve the purpose.

hope this helps
s.arun prasath

web services have become far more mainstream in the past years, since they offer a standardized method by which systems can interact. We developed a web services module for our integrated BaaN portal solution. Initially, our portal was designed as an e-commerce / supply chain solution to allow human users at trading partner's to have a secure / simple interface to perform real-time inquiries, transactions and generate reports right out of BaaN, via point & click GUIs.

As an extension to this, our SOAP/XML web services provide a simplified method to allow any of these things via DIRECT systems integration. Furthermore, WSDL's (web services descriptive language clients) even go as far as virtually automating the integration process itself, making it easy on IT resources at both ends. We've created web services which bridge our client's BaaN with customers, suppliers, external e-commerce sites and even internally with shop floor data collection suites, kanban packages etc... whether they're .net, java or pretty much anything.
