I will try to answer the ones which I think I know.

2. You will have to re-write client side BOIs, but not the Baan side (if any).

3. Baan Portal was a re-branded version of SAP portal. For more information on the product, you can look at the SAP site.

6. OpenWorld has standard adapters mostly for Baan products, such as EEnterprise products. They now have standard adapter for SAP as well. Custom adapter can also be built for other applications.

7. QCD stands for Query Content Description and DCD stands for Data Content Description. These two documents together define a client side BOI. QCD holds the query that will be used and DCD holds the structure of the data that will be retrieved. The BOI compiler generates the java code based on these two documents.

No. By client side BOI I meant the BOI that is present on the client application. Depending on the requirements, a BOI can have a client side (that will reside on client application's machine, e.g. ESales) as well as Baan side components (which will be in the Baan Server). By switching from BCBE to OpenWorld, I think, you will only need to recreate the client side components.

The Cordys tools generate function servers to expose the logic of sessions. Using SOAP, you can do any insert/update/delete in any session. What we have also done is call our own DLLs, for some custom action such as "complete production order". So the existing sessions don't have to be modified.

Reads are done by going through the BDB Library.

So this means that the client UI has to be redone in HTML/JavaScript, Java, Perl, or VB or anything else that can send/receive XML over HTTP.

That's how it is for Baan IVc4. Apparently the newer Baan Connector from Cordys can optionally uses the OpenWorld JVM hooks, but I don't have any experience with that.
