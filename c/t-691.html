Steve Johnson
7th January 2002, 18:24
You do not need to buy anyhthing to launch Excel or Word unless you want capabilities beyond what Baan offers. Set up a printer as follows:
Device: Excel
Device Type: Rewrite File
Locale: (usually ISO-8859-1, but whatever you normally use)

Driver: (blank)
Shell command: (blank)
4GL program: ttstpconv
Argument: ASCII:excel %s
Path: (whatever, but something like ${HOME}/baanrept)
Page length: (wahtever, but something large like 100)

That's it.

Word output will be simply text. You can Edit/Select All and change the default font to something like Courier to columnize the data.

Excel reformatting takes a couple of steps, but not hard:
Select colunm "A", the first column. Select Data/Text to Columns,
Delimited/uncheck Tab, check Other (put pipe symbol "|" in Other field), Finish.

If the file just gets to the PC in the same form with delimiters and you import that into Excel/File/Open, Excel will automatically open the Import File Wizard (aabove) and format the file on the fly. Invoking Excel from Baan as a device will not and you have to do that manually. Also note that every 9say) 100 line is an unwanted page heading. This inconvenience is taken care of with products such as B2WIN.

2nd April 2002, 16:40
Another "trick" is to set the 4GL argument to

"ASCII: start excel %s"

This eliminates any need to change the PATH system variable on the Client.
A few Clients that have "Paintshop Pro" will not work with this setting, but everyone else seems to work fine.

There may also be an accumulation of "tmp" files in the Client's Baan folder. If a file name is configured in the device setup, then a single file is overwritten each time. So any 2nd attempt will fail while the 1st Excel session is still in use. If none is configured, then unique temporary names are used and the users can have more than one at a time.

My old problem of not working when there was a space in the path name to the PC file was resolved by using a newer version of "ttstpconv".
That was on BaanIVc4.
On HP-UX I used "what $BSE/tools/ttB40_c4/ottstp/ostpconv" to see that the second line of output showed the date of that version was "98/12/17". We changed to use a newer version that showed a date of "01/08/29" and the problem went away.

If it does not work...
Open a DOS window.
type "excel"
Does Excel start up?
type "path"
Is the folder where excel.exe is located shown?
If needed, modify PATH=... in autoexec.bat or wherever else it is set.
Even if Excel does not start, the file should be on the PC in the Baan Client directory. If not, then something may be preventing the report file from being copied to the PC.
