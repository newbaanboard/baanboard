The DDC is made to run on handhelds. There is overhead for handheld menus and other handheld field control features built into the DDC. The DDC requires the middleware ddc driver to function. That piece bridges between the Baan DDC module, which is message/transaction based, to the handheld. You can purchase one as you mentioned, or attempt to write your own - Baan has published DDC Middleware standards to start from. My company does a lot of DDC's and a lot of other types of coding as well. In the past, we have used the ddc to interface to 3rd party products, even if it's just to use the DDC Function Server Generator to get nice 3gl versions of the Baan 4gl code. The DDC Module provides good logging and testing capabilities. However, with the introduction of AFS, I would not suggest you use the DDC module to do anything but interface to handhelds or the palm pilot styles interfaces. With AFS coding you can call the Baan sessions without the source as needed in the DDC's. (You can combine the DDC with AFS coding to get around the source code needs of the DDC. But still, the DDC has a lot of overhead directed specifically at handhelds, and the middleware is an expense or a big project.)

There are lots of ways to call the AFS's from outside of Baan not using the DDC. Some free, some not. Off my head, I know you can call Baan libraries (AFS code) from Visual Basic using OLE instructions, and from Excel. There are samples of this type of coding on these boards. You can go up the $$ scale with interfaces such as Cordies and OpenWorld, which provide development environments, testing and logging capabilities for you.

Hope something in there helps - good luck!

Hi,

If you're C oriented like me you can use C# to interface with AFS, you can turn the Baan Type Library into a .NET assembly and use it like a class.
I also had to look into making handheld wireless scanners work with Baan (receptions).
After some investigation and reading through this forum and also with the help of the gurus out here (thank Marks and others) I decided to go for AFS and I'm happy I did.
I used Wavelink to interface with the wireless scanners (we have different models) which also provides a COM interface. So I developed the wireless interface using C#, a console program that talks with Baan&handheld using C# (this was needed because the OLE object opens the option window which in turn created problems with the wireless scanner) and the Baan DLL.
I runned into all kinds of trouble with it (especially making the receipts AFS work) but it's working very well now. Also by starting with receipts I got some experience with AFS and have done more function servers without many problems.

I would go for AFS because:
a) You can use VB/C# and easily interface with DLL's you create inside Baan.
b) AFS simulates user entries so you get all the standard validations.
c) Sooner or later users will want to automate several Baan functions (fill SO from file, fill PO from file, etc) so it's a good investment to learn AFS.

Hope that helped,
