If what you are looking for is the estimated consumed hours that would appear in the order's estimated costs, then I think the calculation for a single operation is:

((cycle time * order quantity) * (man occupation for production)) + (setup time * man occupation for setup)

I think there is a parameter that indicates if setup time should be included in the calculation.

If the operation is flagged for fixed duration, the cycle time is used in the above calculation but not multiplied by the order quantity, making it:

(cycle time * man occupation for production) + (setup time * man occupation for setup)


If the same calculation is done for each operation and summed, the total should be the order's estimated consumed hours. If elapsed time rather than consumed is needed, the various lead time elements such as queue, wait and move times can be set up; these add to the duration of the order but not its estimated costs.

Jeff Ball
25th February 2014, 14:56
I think that only phantom components will roll-up operation times into the end item. Normally, an end-item's component costs are considered to be rolled up into the material costs for the end item production order. LN has the option of cost component charts that allow component costs to come across as a list of cost components, but I think in 5.0c they are just included in the material value. So the end item estimated hours should only be for operations within the end-item's routings (including phantoms)
