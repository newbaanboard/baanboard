the mappping tools are fine, but they promote the idea of filling your nice new LN system with all your redundant BaanIV junk.

I would treat this as a reverse archiving exercise and only migrate what you need, not loads of obsolete parts, orders etc. IV to LN is a big step with lots more stuff to setup than the BaanIV data can provide - plus do you really just want to implement LN with the limited structure of your BaanIV system?

LN exchange module with DAL is surprisingly good for imports and generally prevents inconsistent data (unlike exchange in Baan4) - plus it allows you to be more in control of your project.

Have done a lot of B4 to ERP LN migrations, a few with infor data conversion adapter, but most just with a few scripts with 99% dal functions and 1% afs. Using your own script is more flexible.
You can recode a lot of base data (mcs data) and you can clean all static data items/routing/bom/ etc. If needed you can create one archive company in ERP LN which has all old orders / items / projects etc. But the life company is clean with only the items you want and only open orders, open invoices / inventory etc.

Good luck with the conversion project!

We've done a few too; if fact we're doing several upgrade projects now. I agree with Mario that scripts are more flexible. I would suggest that if you use the Baan 4 to LN conversion tool that you exercise caution. It's not 100% and has to be "hacked" (i.e., GTM) in certain areas. This is particularly true where there are new tables, new fields and new functionality.

I too would try to dissuade you from migrating history. Archive this and keep it in a Baan 4 company read-only or a data warehouse. When taking across master data you have an opportunity to add prefixes to keep different types of items, business partners, etc. separate. You can do recoding of items but beware the links in tables such as tdsls401 Sales Order Lines, tdpur401 Purchase Order Lines, etc. when you're migrating open orders because of the references.

One thing that we've found causes headaches if you're not prepared is when you're adding new capability that requires data changes. For example, when you turn serialization on then you have to set flags in tables like tcibd001 General Item Data and migrate serial numbers. Then you have to reference these new serial numbers in open orders.

Again, good luck with your migration and let us know if we can do anything to help.
