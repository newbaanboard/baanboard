Hi Eduardo,

if your inf_storage is so long, that it requires automatic sorting, then you should consider to restructure your setup.

A very long inf_storage has a negative impact on performance. When the bshell starts it wil read the inf_storage only once into an internal array. But for each table access it has to consult it and look up the corresponding settings.

It is advised to store the inf_storage file only on one server, from where table creation and deletion activities are taking place, since this file will get loaded in shared memory and it will occupy a lot of memory.

Two things:

1 - Sort it however you like, but your "catch-all" entries need to be at the bottom of the file.

2 - everyone should have large x_storage and small x_storage2 files.

x_storage2 is loaded into bshell memory and should be 2 lines (based on what your driver parameter manual says).
