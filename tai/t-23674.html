Hi chadwickh,

there are a couple of issues, which your approach:

a) having such a session, would anyway just test batch performance.
Testing batch performance is important for each migration, but for this you can take indeed a standard session like the MRP or a statistics rebuild, which are repeatable.
If you use such a batch test, then you need to do this anyway in something like a donwtime period, because otherwise the result is messed up by concurrent user actions.

b) The real issue in testing performance is actual online performance of X concurrent users. This is the real challange and a single session will not help you here. Especially in your situation (going from a HP to a windows box) scalibility by CPU count will be affected.

c) Finally you have to consider, that the performance is always much more dependend on the specific code executed. A silver bullet solution to increase performance does not exist. A silver bullet solution to measure performance is therefore also nonsense. If the performance for your test session is good, this does not imply that the performance of other major sessions is good, as different SQL statements are executed.

But I agree that a performance testing mechanism would be of great help for any migration. So maybe somebody here a solution without the pitfalls I described?
