There are about 40 tables which need to be audited and are manditory for Baan to run. You will see these if you look at Maintain tables by Database and see that most of those are parameter tables.

Other tables that should be audited are totally up to your descretion. Uusally auditing is only done to facilitate data exchange (i.e. using exchange with audit). I have been at over 40 clients, and none used auditing as a security measure or needed to keep a handle on what fields changed and by whom.

If you really need to audit tables to ensure some users do not modify data unwittingly, there are other methods of appling security by session, company, table, field or field data by value which can be used. You will find that functionality in authorizations,

I would look at auditing at a last resort since it can produce a lot of log data and can slow down your system if you audit too many tables. I would only use audit in concert with exchange.

Sometimes the accountancy-companies, the ones who actually "audit" another company, have specific data-types prescribed that must be audited. For example banktransitions. So there are three criteria to consider when you implement your audit-trail:


BaaN
As already mentioned: a number of tables must be audited, otherwise BaaN won't run at all.

Accountancy
Accountants wish to be able to see (most of the time annual) reports of who changed specific data. The list of tables to be audited should therefore be discussed with the specific accountancy-company.

Own wishes
Which data is crucial for your specific company? This will differ per company, ofcourse.
