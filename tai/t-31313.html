It has to. If you have 10 different companies linked to the same Pack Combination you could potentially have ten different sets of "Job Data" tables (you know the ttaad5XX). That means that nothing prevents you from having a job with the same name on each company. The only way to differentiate them is by linking them to their companies.
Even if potentially it could have been implemented as one daemon polling all the associated companies it is not a good idea. The biggest problem job daemons have is that it is a single bshell attending the requests of all your jobs. That is terribly inconvenient as it is right now running all the scheduled jobs in only one company. Imagine letting it handle those of all your companies in your Pack Combo. One bshell means only one session will be run at a time. So for starters you can't run simultaneous jobs. But the worst part is that one sessions that hangs your daemon will cause all the next jobs to fail.
Don't use job daemons. Cron for UX and Win scheduler for Windows work much better.
