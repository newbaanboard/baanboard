As for the observations: the file on disk is being updated.
Then there is one option that needs to be checked:
when you first update the ttstpsessinfo session (using solution 126983), you can then run ttstpsessinfo on the session with the Object Info checked.

NEW FEATURE! you will see what report from disk is taken.
Here there are two options:
1) The ttstpsessinfo report points to the correct file on disk
2) The ttstpsessinfo report points to the wrong file on disk

1) Now all seems to be ok. But still the correct report is not used. So apparently there is something in between. This may be shared memory or OS caching. That is the problems where I referred to in the first reply. In this case: report what happens when the system is rebooted.
Also (first) try to run the session with the options
`ba6.1 -- -keeplog -logfile readme -dbgsrdduse -dbgfile tdpur4401m000`
File readme will contain a logging of files used and what files are from shared memory. Then try to do the same after reboot. Compare the files.

2) Check the developers data (VRC) and user data (ttaad2100m000) for the correct derivation structure. Apparently the developers data point to another VRC which is not (first) in the derivation structure.
This can be the case when you compile with the developers data into the tdB40C_c4_ABB, but there also resides a opur440101001h file in a higher (Own) layer in the derivation structure. Actually explode6.1 and bic_info6.1 should report on that file instead, but that may be an error (starting to search where the report is in tools?).

On the ttyeno problem - I expect that problem to be solved in newer software version. You did not state where that error came from, if from ttadv3240m000 or ttadv3130m000, then search for newer software on those sessions.

I can also imagine that - because you use language h - the $BSE/tools/dd*/dtt.pd/dtt.pd does not have an entry for that domain in language h...
