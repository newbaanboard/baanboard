Installing a solution only when something breaks is very similar to gambling. You're gambling that
1) nothing breaks
2) if something breaks there is an already published solution that will fix it.
3) the solution and it's dependencies won't cause more problems.
4) the thing that's broken isn't needed "RIGHT NOW".

In a small environment, that's probably acceptable and may actually be the only way you can go.

Effectively:
(The cost of installing a patch) > (The cost of not having a critical business process for the length of time it takes to patch, and any additional process that break due to the patch)

Unfortunately, most IS managers feel that way, additionally in many of those environment the following is true as well:
(complaining about Baan) > (treating baan like it's an integral part of your enterprise)

In fact there is a considerable risc of breaking a running system when installing all solutions blindly. So, I would not recommend that.

With LN, I would recommend installing the most recent feature pack every 2-3 years (depending of the customizations you have - the more standard sources you modified, the less often you will install feature packs, since its some work to do).

A new feature pack is a major test point that you have to treat like a new software version. Your users will have to extensivly test their processes. So, install the feature pack and solutions for this feature pack up to two or three month before the current date. Not *all* solutions that are avaiable, since the newest ones may contain bugs. Install like this, test, and if an issue arises, install only the solutions you need to resolve that issue.

Now, assume you install a feature pack every three years. Installing a solution a year after the feature pack may already become cumbersome. So at least after 1 1/2 years you should do a test installation of all solutions currently available (but again, not up to date, but up to two month before now), let the users test(!) this patch level, install additional solutions as needed and then switch the live system to that patch level.

So, I would recommend to have a basic patch level that's not older than 1 1/2 years, upgrade that level every 1 1/2 years, and then install solutions only when needed. I would not let this basic patch level go much older, since otherwise, when you urgently need to install a solution, you will have so many depenend ones, that you basically will change the whole software without being able to properly test it.

Hope that helps

Dave has good points.

We install the weekly patches regularly, and porting sets every quarter. If there is a bad patch the probability of finding it and uninstalling it is manageable when you have a good environment upgrade path (Sandbox -> Dev -> QA/Test -> Prod). If you find an issue with particular patch in sandbox you can open a Support ticket and work with them and retest a new patch before you apply the bundle to the other environments.

If you install all the patches in a big bang fashion, then its hard to isolate which one was the bad patch to uninstall. Doing the regular patching also reduces the number of pre-requisites and downtime during the Feature Pack upgrades.

Note - The above statements are referring to the Tools (tt, ta, tl) patches/feature packs.

Yes, Tools patches are somewhat different. I agree they can be installed more proactively, the risk that they break something is not that big (since they contain most of the time just a few objects, and since the tools layer can be tested much better by Infor than the application, using regression test suites etc.). Also, most of the time they do not involve reconfiguring tables (which makes a solution hard to install and hard to rollback, too), and the dependency graph is more or less linear (not a tree as with the application)
