The rule has changed somewhat to be more consistant with standard SQL. All the parts of an index must be defined for the refers to to work. Also, the as referred by and path constructions are iffy now. Here's a link (http://www.baanboard.com/baanboard/showthread.php?s=&threadid=9223&highlight=index+parts) to the board discussion on the refers to changes, and the Baan explanation.

This link (http://www.baanboard.com/baanboard/showthread.php?s=&threadid=5707&highlight=referred+by) (last thread) also has relevant information.

As for upgrading your custom code - that depends on how heavily you relied on these statements. I can say that when this hit us, having developed a LOT of custom code over the years for people, this only effected a couple sql statements, and was easy to fix after understanding the changes.

The Baan code has been upgraded to conform to these standards.

I will modify the above after these years to say that the major change is on the 'referred by' clause, I have not see the 'refers to' to cause the error. So, if you search for 'referred by' in the scripts, you should see pretty quick the ones affected.

Here's a Unix search that I use to find pieces of code:

find . -type f -print | xargs grep referred by

. = starting directory - start this from $BSE\application
-type = type file
-print = print results
xargs = use grep command on the files found
grep = find something
referred by = what you want to find.
