Password aging can be implemented in a Baan system that resides in UNIX very easily (granted the UNIX administrator already enabled it for all the users at the OS level). There are two relevant solutions that need to be reviewed on how to configure password aging within the application to enable it.

Password aging in Baan is only responsible for:

a) Allowing password change granted there are no restrictions on the account, e.g., account is already expired, or password was changed already and there is a rule that says at the OS level that users can only change password once a day, and so forth.
b) Notifying users if the account password is about to expire and give the option to change password and allow the use of the session change password if the user desires to do so, granted there are no restrictions on the user account at the OS level as exemplified above.

The rest of the tasks fall directly on the UNIX administrator, Baan password aging will never allow a user to unlock his or her own account. That is a task only the UNIX administrator can do.

The two solutions that need to be reviewed under infor365 - Downloads - BAAN / LN - Direct search are:
a) 119781 How to activate password aging for UNIX using the Baan GUI
b) 117156 Correct UNIX authorization of the binary badmin6.2


Like nprao mentioned, all this functionality is built around PAM so the password aging object makes calls to the binary (Badmin6.x) that subsequently makes all the PAM library calls and these libraries either validate, authenticate, verify users accordingly.

PAM is implemented in every flavor of UNIX differently and also the libraries to use will be different depending on the authentication mechanism, LDAP, Kerberos, passwd, etc.
