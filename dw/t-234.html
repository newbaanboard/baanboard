Jeff Ball
10th October 2001, 19:25
More than you might think...

There is a new version of the Hybrid Logistics model

The linkage between the components (BCD, Function and Process, goes back a little towards Baan IV - the Function bubbles are back, and the use of case components is being abandoned

There is no longer a capability to do organisation modelling in a diagrammatic form

'Helper Applications' have disappeared; instead there's a new concept of Components and Applications to configure Baan and non-Baan executables

The method for generating user desktops has changed somewhat - its complicated to set up but very powerful in controlling access

The entity relationship modelling now works well, with automatic generation of links for any entities you might wish - its grrreat!

I'm sure there are other bits and pieces, but I haven't stumbled across all of them yet...

Your response leads me to believe that you're not using Subsession Authorization. Here's an explanation ... and hopefully, I'm not telling you something you already know!

Subsession authorization is a detailed way of setting the authorization level for a session that can be accessed from another session by zooming or some other means.

For instance, from tcibd0501m000 General Item Data, you can double-click on an item and get to subsession tcibd0101s000 General Item Data. From there, you can zoom out to tcmcs0123m000 Item Groups ... and from this session you can create a new Item Group.

Well ... maybe you don't want some of your roles to be able to create a new Item Group, so in the Repository, on the activity box for tcibd0501m000, you insert subsession tcmcs0123m000. You can leave the authorization set to Full Authorization because you want some of your roles to be able to create new Item Groups. Then in the Project Model you open the subsession session by clicking on the Subsession icon, and you Edit Roles. From here, you can add the roles that you want to give 'No Authorization' or 'Print/Display' authorization.

Of course, if you didn't want anyone to be able to create an Item Group from the General Item Data session, you could set the subsession to 'Print/Display' in the Repository.

I was just wondering if the procedure was still the same in 5.0c. because the process of adding all of the possible subsessions to an activity box can be quite time consuming! Especially if you have hundreds of processes ... the way we do.

Hi

Well, this time you have got a true, but very limited answer. The roles are defined not in the project models, but in the repository. It is the employee code that you append to the project model. I think that's the same in 5.0b...?

A difference that you may not be aware of is that it is no longer possible to model org diagrams in the DEM Editor, but the role, employee and employee group functionality is still there.

Thus the list of roles that you define within a version is available jointly for employees in any project model in that version. I have faced a similar issue on my curent project; we have a single finance, multi-logistics setup on a shared server. The roles are centrally defined and (thus far) consistent in their process mapping. But when we come to build post definitions from the roles, the different sites vary in their understanding of what a job really means.

We have devised two ways of overcoming this, both based on use of versions. The first step was to determine that each site would not only have its own project model, but that the project model for each site would sit in its own version, all derived from the common version where the process repository resides. The first method was to define those roles which have variant functionality between sites, and copy them down into the versions where the project models live. The employees linked to the project model had this as their default version, so since the role to process mapping can be different by project model, employees logging into different logistics companies get their correct desktops. The other method is simpler, and works well when job boundaries differ between sites. Then we can leave all the process-linked roles in the common version, and create 'super-roles' in the derived versions to simulate post titles. The employee is then linked in the derived project model to a single role in the site specific version, which maps to one or more roles in the common version.

Sorry this has been long-winded, but I hope it helps you get over the 'can't do that' answers which sound more like 'don't fancy the work involved'

Happy New Year

Jeff
