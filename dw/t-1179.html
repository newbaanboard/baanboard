There is no simple answer to whether to use DEM or Tools to generate authorisations, but some general guidelines are:

- if your DEM flows constitute PROCESS INSTRUCTION rather than just being a collection of relevant sessions, use DEM. This is because changes in access should only follow changes in procedural instructions

- if your role definitions are rigid, incline towards DEM. If your roles/job titles are based around individuals' capabilities, use Tools.

- if a number of people with variable skills are going to be involved in administration, DEM is helpful because it is much easier to work out WHY a user can or cannot access a particular session, and the skill level required before allowing someone to make changes is lower.

- Avoid Desktops. Thay are seductively easy to set up, but anyone that can remember Run Program / mtccom000000001, or mtccom000000002 etc can get back to the full menu browser, so your Tools authorisations still have to be watertight.

- To avoid the problem common with DEM of how to access all the things no-one thought about during process design, just add a new process, call it General Enquiries, Miscellaneous, Aladdin's Cave or whatever you want. Then add lots of activities, and link a utility to each one, and set up as the contents of the utility all of the print/display sessions relating to each topic. Make each one print/display access only, and attach all the roles. It will take 2-3 days, but then its done forever!

George,

The best way to handle these issues with minimum maintenance effort is:
1) Define what roles and subroles you actually have.
2) Build the matrix of roles and subroles
3) Evaluate your model, do the flows truely represent work activities??? By this I mean they are short, concise specific work assignments that real people perform.
4) Rebuild the flows where necessary to reflect 3) above
5) Perform a where used sessions in the model. Look for instances of sessions which are repeated more than three times. Theses should either be placed in a utility or subutility or your design of the flows using these is inaccurate.
6) Assign the roles and subroles to the amended flows
7) Attach the utilities (main and sub) where needed

This should allow you to have the greatest flexability with the least maintenance effort. If your company needs specific detailed help, I am a certified architect in DEM and have 25 years BPR experienece in over a dozen vertical industries.

To accomplish this requirement is trickier and takes much more effort.

1) You would need to identify restricted, normal and super user roles. You know what normal and super user already are defined as in Baan. The restricted role (a normal role where you have eliminated accesses by session) would have access to selected Baan Maintain sessions and subsessions, as defined through Tools. By not giving access through Tools to the sessions you want to restrict the user from accessing; in the Maintain sessions, they could run the zooms and links but could not access those zoom or linked sessions from the session they were in.

2) These roles could then be assigned as any other normal roles to the Process flows and the roles to the user. This would restrict their ability to access via zoom and special what they could access.

3) This use of the "restricted" role could allow you to give your users a similar look to their menus, but the ability to enact actions would be different, depending upon the role combinations between normal and "restricted" you have assigned.

4) Each "restricted" roles could be identified by adding an r to the front or back of the role label to distinguish it from its normal counterpart.

Not easy, but doable. It requires a premapping of these sessions against the links they have by zoom and special. I would strongly recommend that the mapping by session you do to set this up be retained for use when you have a need to modify these "restricted" roles. No need to reinvent the wheel over and over.

> Ensure the same menu for every user:
- This should typically be handled by DEM. Assign the proper roles and generate the menus. If one of the roles changes, all usermenus using that role need to be re-created.

> DEM is managing authorizations 'by default'
- This is a good way of handling access, although some EDP Auditors may not like this 'leak in security'.

- I normally set up user templates in Tools that have the proper authorizations. These templates have a 1:1 link to the Roles!

> Authorizations on sub-sessions
- In your case you do not set up user templates to 'include' the session-authorizations as you let the user open the sessions from the menu.
You may set up the user templates for 'exclusions' in the session authorizations. This way some sub-sessions are not accessible.
But as always there's also a problem with this way of working: if one role 'includes' a certain session and another role 'excludes' the same one ... and one user has both roles ...

You still have some decisions to take, good luck!
