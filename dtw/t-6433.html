Sorry to correct you, but Baan IV behaved exactly the same. The Adress is is master data and is only linked to an order. The only exeptions are order specific adresses.

The behaviour you expect would require the software to copy the actual adress data each time it produces a new document. That would create a huge amount of redundant data and is therefore avoided in any transaction system like Baan.

The "issue" you describe happens for example also with item texts. If you change the item text, you will change it on all previous invoices that used that item. The same happens when you change the descriptions of delivery terms, payment terms, whatever. If your order references some master data (and adresses are master data), changes in that master data will appear on all already printed documents.

So, from your controlling point of view, reprinting a Baan document number like an invoice is not "document safe". But that is not indended, too. Reprinting is a functionality for documents scrabled by the printer, or for information of employees and things like that. If you need a document save archiving functionality, you have to either archive the paper or use a (third party) document management system, that stores the image of the printout, and not the relational data.

The reason the address changes for the existing orders is because the address is only linked to the order. The address information is not stored on the order itself! So, if you change any address, the new address information will be used on any report for any order that is linked to that address code. This is just something the end-user needs to be aware of. It is strongly recommended to create a new address rather than changing an address.
