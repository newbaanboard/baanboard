Not sure which version of BaaN you are using but I know under IVc2 there is a "bug" regarding the Maintain Relation by Order Line session. Anytime you run this session from within the sales order, BaaN looks for the default relation and automatically links it - if there is one. If you delete what is there and do NOT replace it with someone else (in otherwords leaving it "blank"), leave the session and then go back into it once again, it re-links it back to the default relation! So you can never really use this session just to "review" what is linked to the line as it may alter what you want to be there. This problem has been confirmed with BaaN support and there is no fix for it in BaaN IV. If you need to review the info, better to look directly at the table itself (tdcms041m000).
We were going nuts with this trying to figure out why, after deleting relations manually from a sales order line they were reappearing! BTW, if you manually insert a different relation from the default one, BaaN leaves it alone. It is only when you try to leave it blank that it keeps filling in the default.

we are fighting commission and rebates since almost four years now and still find some unexpected results when running the calculation.
A warning for all those who are deleting salesorders (Baan IVc4):
a salesorder with total amount = Zero and an automatic link to a relation never gets calculated (standard functionality). When you delete salesorder and salesorder history the link between order and relation will not be deleted! In case you use the order series again the new salesorder is in a way 'prelinked' to the wrong relation. The automatic link process doesn't generate an error message like 'Duplicate Value in tdcms040' or something like that.
We were lucky to discover this effect just before running the final calculation for a large number of salesorders.

We have done a couple of things for different clients. Like Martin, we created a clean up program. We also modified things to allow $0 commissions. This was motivated by a client requirement to allow pro rata (percentage) splits amongst one or more relations (e.g., relation A gets 25% of the commission and relation B get 75%). In some cases, they want to know that a person is associated with a commission, even if that person's share is $0. Overages (additional commission based upon the "book" price versus the "sell" price) also complicate things for some clients.

I think the concept in standard Baan is OK, it just requires some tweaking (OK, sometimes you need to tweak it with a sledgehammer;) )
