We have just implemented Bar coding using Lot Control.

We are having some issue with Outbound and was hoping somebody may have some suggestions.

We generate our outbound advice, which in turn creates suggested lots on the advice line.

When we pick the goods we dont necessary pick what is suggested by baan. We select the first advice line and change the lot from the suggested to the actual and can also change the quantity.

If this fulfils the total requirements we then carry on to confirm shipment.

But in back ground on the outbound advice line the other suggested lots are still allocated on the stock unpicked (giving us problems when trying to ship these out on other orders).

At present we are having to manually go back into the advice highlight the suggested lots that have not been picked and delete them off to free up the allocated LOT number to allow us to ship out on other orders.

Has anybody else come across this problem using lots? :(

Sandra,
We addressed this issue in Baan IV, more that 10 years ago. Actually when we picked the lot we do lot swapping. We used to check if the lot that was picked was on other order, if so swap it with the order, else then it is assumed that the lot that is picked was not allocated. For this we created a index in the outbound table with Lot and Item (and other fields) as index.
I don't see any other way. But if your requirement is to track the lots shipped to customer, then probably you don't need to implement lot control in Baan. In my current installation we only keep track of lot that are shipped, and the same info is captured while shipping (by our shipping systeM) and is transferred back to Baan and we store that in additional table.

We do both Roll to order and stock goods.

We have for years been using an 'advisory picking list', a report we created in baan. They use this to pick the goods and then we use the picking/shipment stages in baan as the loading/despatch stage.
