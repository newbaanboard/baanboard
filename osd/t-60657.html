The Linux OS and Oracle DB we use for Baan IV c4 are going out of support and I am considering moving to Windows and SQL.

I've always used Baan with Unix/Linux and I wonder what sort of issues I will face moving to Windows.

We would probably be using the latest versions of Windows server and SQL on a 64bit platform. We are not considering upgrading from Baan IV at this time.

Why am I considering Windows and SQL?
Better integration with our Windows network, printers, backup system, single sign on, etc.
No annual support fees for SQL or Windows (unlike Oracle and some Linux platforms).

One main question I have and can't find answers for is can users access Baan on Windows through an ASCII terminal emulator? Some of us use the GUI, but the heavy data entry people prefer the ASCII. They can enter massive amounts of data very quickly in multiple sessions without ever taking their hands off the keyboards. Loosing that productivity by moving them to a keyboard and mouse would be a tough sell.

I know that if there are scripts that access the filesystem, I need to be careful with the slashes when moving from Linux to Windows. We only have a few cases where that would be an issue. Are there any other programming gotchas I need to be aware of.

Thanks,
Rob

We had migrated from unix to Windows about 4 yrs back . Nothing special as such and u may probably know many things .

1. 64bit does not have any memory limitation . So go for higher memory as far as possible . SQL is very resource intensive. But with hardware and resource upgrade , we experienced tremendous throughput improvement.

2. All porting set binaries are exes as against 6.1 in unix/linux.

3. When doing installations / other baan admin jobs on server using RDP , take it in console mode.

4. All programs work without modifications barring some SQL queries involving double comparision of odd figures worked some times . So we had recoded those part. But that was very minimal and was detected and resolved hardly in couple of days.

5. Baan installation / config documentation is quite comprehensive and try to follow it scrupulously.

Basicly you can move your application to windows without a problem. However:
- ascii interface is not available;
- you can't use bisam as database;
- printing is done via windows drivers, it looks a bit different, you will get some comments about it;
- make sure that for the printers you use windows drivers are available;
- check used path's in jobs/scripts etc, could be you have to change it;
- scripts around the baan environment will not work unless you use e.g. cygwin but still will need some rework;
- if you use multiple baan environments on one server be warned that updating a portingset will bring all environments down;
- you will need to export all data and import it again via bdbpre/bdbpost. Oracle is less strict in date data so you probably have to repair some data before being able to import into sql.

Regards,
Han

It seems to work ok but never having needed an ascii screen interface, I don't know if there is any functionality missing.
