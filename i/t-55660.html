Hi,

I can't give you a specific solution but our experience may help you to narrow the focus before you need to contact Infor Support. What Feature Pack of LN 6.1 are you using? Are your patches/solutions up to date? If you enter from/to ranges for Integration User Group, does the process complete successfully (that is, maps and posts) or does it fail to complete but just does not result in an error message?

I recently (Jan-March 2009) had a similar situation during initial testing and validation with a client that was implementing LN.

In our case, the primary difference was that the implementation was Single/Single, not Multisite. We were running LN 6.1 FP3. Patches were presumably up-to-date but I did not personally check to confirm that all relevant solutions had been installed.

When we ran Map Integration Transactions with "Post" also selected and with all selection options set to wide open defaults, the session errored out. It did not kick us out of LN (i.e. did not end the bshell). I did not save a screen capture (see next paragraph) so I'm not certain that the error message was the same that you have documented. I do recall that if I entered values in any option range (for example, the first and last valid IDTs, not just Integration User Group), the session/process completed successfully. I think that when I ran the "Map" and "Post" sessions separately, I did not get the error but my recollection may be faulty. I did not find any "helpful" entries in the error log.

At the time, the client was not not yet in production and the error was an inconvience, not a show-stopper. Before I took time to make screen captures and log an incident with Infor, the client upgraded to FP5 and installed all patches. After that, the error condition did not recur.

I had not previously encountered that error condition in other implementation of LN 6.1, SP1/FP1, FP2 or FP3 (including single/single and multisite) so I assumed it was a bug at some point in FP3.

It is always possible that a localization for India contributes to your specific error but my experience would suggest otherwise.
