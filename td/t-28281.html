AFAIK , there is no event triggered on an input field as u keep on typing . So executing a search code for each key press on an input field may not be possible in Baan Iv . Dont know about the later versions.

Also from performance perspectives , such feature may be slow and it can be more useful for small database with VB kind of front ends (where client is executing the code) that too for an indexed fields only (unlike description) . In a client server ERP application like baan this may be difficult to use even if there is an event triggered at each key press on an input .

There is a new feature in the Reger, Gemini/LN Tools. Refer to -
fields.autocomplete() (http://www.baanboard.com/programmers_manual_baanerp_help_functions_form_and_form_field_operations_fields_autocomplete)

fields.autocomplete()
Syntax

void fields.autocomplete( long try.autocomplete [ , string field_name(18) ] ... )
Description

Auto-complete is automatically executed by the 4GLE for fields that have a reference path to another table and have a zoom session and has no button and is not segmented. With this function auto-complete can be turned off ( try.autocomplete = FALSE ) and on again ( try.autocomplete = TRUE ).

For array fields, the element number should be sent as part of the string, between brackets. See example
Context

4GL library function.
Example

before.program:
fields.autocomplete(FALSE, "ttadv200.cmod", "ttadv200.cmop(2)" )

I can think only of one way, but depending on the number of items you have in your table, you will always notice serious performance slow-down...

For my idea you need to have an ascii file somewhere on disk (for instance ${BSE}/items/items) containing all the item descriptions the same way as they are stored in the Baan table. During runtime you create an object (function create.object() ), which will be the form field (so no field defined on the form), and in this field will be listed all the descriptions. Then you can just start your input and have the autocomplete functionality of Windows. In another form-field next to the description, you fill the item code (which you fetch via a query on user's selected description).

It is a bit a different mindset, but it must work. I didn't try it now, but if you need assistance I can try to give you some conceptual example of the code I suggest.
