Refer to BaanERP Application Software Engineering Guides

Marking changes in scripts
--------------------------------------------------------------------------------
During development of new scripts or a complete redesign, the number of changes will be large and there is probably not a need to indicate individual changes. In this case tracing changes via the checkin/checkout mechanism can be sufficient.

The decision to mark changes is more or less decided per project based on the following reasons:

Planned merge activities to other package VRCs in a parallel project environment, so the individual changes have to be interpreted individually
Requirement changes
Customer calls, e.g. Product Maintenance
Below the markings for functional adjustments and error solutions will be explained.

In the lifecycle of a script multiple ways of codes for marking can be used:

During development topic code, Business Requirement number, TCS call number etc.
During maintenance, the (Scopus) solution number
A defect solution or a brief topic description requires text at the end of the program script header. When SCM is used and the same text is entered with SCM revision text, this will be done automatically. In maintenance mode the PMC tool is used for checking in.

For instance (maintenance example):
|* SOL10017, T. Developer, 1996-08-07
|* 1. Lot codes by Unit is not possible, lot codes by Batch not needed (DF12345)
|* So added Lot codes by Unit functionality and removed Lot codes by Batch functionality.
|* 2. Lot codes are not generated correctly (DF23145)

For error solutions add the ident to identify the error solution in the script of DLLs (including Integration DLLs, DALs) and includes. Idents are visible in the object as ASCII text. Idents are either generated by the system itself (BaanERP and beyond), or added manually (Baan IV, earlier)

A example of a manual added example in Baan IV:

#ident @(#)SOL10017, T. Developer, 1996-08-07, B40_c5

Do not change automatically generated ident lines.

Adjusted lines must be marked. This implies that the previous (erroneous) program lines have to remain in the program script!

Erroneous lines on position 1 are preceded by a pipe line | (except for DllUsage and FunctionUsage which is comment).

A the end of such a line the |# is appended followed by the call number or textual identification, followed by .o (means old) or .n (means new).

The identifications are needed because we do not have a revision control system over package VRC's and releases. Source analyzing tools recognize the | character, and distinguish the difference between obsolete and documentend lines for script statistics (e.g. changed code).

Example:

function extern generate.lotcodes()
{
<DllUsage omitted>
domain whclot lot.code

disable.commands("generate.lotcodes")

if lotcode.allowed then
db.retry.point()
if not tcibd.dll0401.generate.item.number(
tisfc001.mitm,
tisfc001.pdno,
| tckltc.batch, |#10017.o
tckltc.unit, |#10017.n
lot.code,
false) then
abort.transaction()
mess("tisfc01204", 1) |* generation of lot code is not possible
return
endif
commit.transaction()
endif
}

If multiple defects are solved in one solution, and the defects should be identified, the solution number may be extended with a sequence number "10017-1.n" related to the sequence of defect numbers in the check-in text and ident string.

If more than one line has been changed, lines must be marked successively. The following construction can be applied where the start is identified by s(= start of marked lines) or the end by e (= end of marked lines). In this example marking during development is done via a topic code.

choice.mark.occur:
after.choice:
if check.order.status.valid() then
enable.commands(MODIFY.SET, DUPL.OCCUR,
MARK.DELETE, GLOBAL.DELETE, GLOBAL.COPY)
else
disable.commands(MODIFY.SET, DUPL.OCCUR,
MARK.DELETE, GLOBAL.DELETE, GLOBAL.COPY)
endif
if tcibd001.unef = tcyesno.yes then |#SER_ITEM.sn
enable.commands("ticst0508m000")
else
disable.commands("ticst0508m000")
endif |#SER_ITEM.en

Use man on what-

NAME
what - get SCCS identification information
SYNOPSIS
what [-s] file ...
DESCRIPTION
The what command searches the given files for all occurrences of the
pattern that get(1) substitutes for %Z% (currently @(#) at this
printing) and prints out what follows until the first ", >, new-line,
\, or null character. For example, if the C program in file f.c
contains
char ident[] = "@(#)identification information";
and f.c is compiled to yield f.o and a.out, the command
what f.c f.o a.out
prints
f.c: identification information
f.o: identification information
a.out: identification information
