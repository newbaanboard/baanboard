Once in debugger, try to issue the command "help" :-)
Use s for step S for step over, c for continue (virtually exit debugging) or q for quit session (plus debugging). Set breaks with: "stop in <functionname>", "stop if <condition>", "b <linenumber>" etc.

you can search in debugger for ".bef.lay" for your before layout sections and set a breakpoint there. Then you simply step into you code. For instance, you can search for the before layout of your first detail section like this "/detail.1.bef.lay" and set a breakpoint. Then you continue to the breakpoint and step into your code. This also works for after layout sections.

If you have to debug a certain layout or part of your report script, you can try to put an new variable into the report like "long debugger.stops.here" and on the script section where you want the debugger to stop, you change the value of the variable "debugger.stops.here = 1". After starting the debugger you issue "stop if debugger.stops.here = 1".

BTW, the debug compiled report gives an idea on how the interaction between the standard program and the session object is working close together. The idea is the same. The session object fills up hooks that are offered by the standard program like the report object does the same with the report sections.

Sometimes you don't know the exact place in code where to set a break point. In that case the feature to stop every time a variable changes its value if very useful, e.g. "t tiitm001.wght" and the just run the program.

The example of OmeLook could be used as "debugger.stops.here = debugger.stops.here + 1" and "t debugger.stops.here".
