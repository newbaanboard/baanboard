While I technically agree with manojsharma there is a product I use called qkey. It allows me to make changes to standard sessions while not actually changing the Baan code. There are other products available - see the code and utilities forum for ~Vamsi (http://www.baanboard.com/baanboard/showthread.php?t=559) code extension tool and RMC also makes a product that does the same thing. I like this method because you always run off the current Baan code. The only draw back is when you run into a problem - Baan always wants you to remove your code to see if you still get the error. So far the answer has always been yes in our case. There is also a sample qkey script in the code and utilities forum. Search the board and you should find more posts.

The replies by Hitesh and me are still valid. I do not like using fields for what they are not meant to be used for. Several reasons (1) Baan may eliminate the field, (2) you may decide to use the functionality the field is for, (3) Baan may decide to use the field. I have seen baan do both (1) and (3) on us, but sometimes users do not tell us what they are doing. So far we have not had any problems. Too many risks for me - but you can make that call yourself.

Here is how a source extender helps me - I copy the form to VRC and add my table field to the form. Then I add code into the form to save whatever the user inputs into the field - usually I use events from main.table.io. So this is one method to do what you want. This method does not require source code and a patch to the session from really should not cause a problem. With qkey I usually just have to recompile the code.

A function server like Hitesh allows you to create your own form for a session with whatever fields you want. Then use just use the stpapi fields to run the session in the background.
